今回はアビリティ設計の方針メモです。コンパニオンキャラクターに自分と同じ技・武器システムを使わせるにあたって、気づいた問題と対処の方向性をまとめます。

## なぜAIにプレイヤーコントローラーを持たせないのか

AIを動かす方法として、「AIにもプレイヤーコントローラーを割り当てて、人間操作を模倣させる」アプローチがあります。これはアビリティが既存のままで動くという大きな利点があり、プロトタイプ段階では手軽です。ただ今回はこの方法を採りませんでした。

| | プレイヤーコントローラーをAIに持たせる | 専用AIコントローラー（今回の選択） |
|---|---|---|
| **利点** | アビリティがそのまま動く・初期実装が速い | サーバー専用で動く・予測なし・軽量 |
| **欠点** | カメラ・入力・UIなど不要な機能を丸ごと抱える | アビリティの依存部分を別途リファクタする必要がある |
| | 入力を「シミュレート」する形になり、AI判断との境界が曖昧になる | （初期コストはかかるが構造は綺麗） |
| | コンボ進行など入力イベントに依存した処理がAIから扱いにくい | AIはASCを通じて直接アビリティを起動できる |
| | クライアント側の予測・レプリケーションが混入するリスクがある | サーバー権限で完結し、予測汚染がない |

AIの意思決定はスコアリングと状態ベースで完結させたいので、「ボタンを押したことにする」という間接層を挟みたくありませんでした。アビリティ側のプレイヤー依存を取り除く初期コストはかかりますが、AI・プレイヤー両方で同じクラスを使い続けられる構造の方が長期的に正解だと判断しています。

## 問題：アビリティがプレイヤーコントローラーに直結している

いくつかの戦闘アビリティを見直すと、内部でプレイヤーコントローラーに直接キャストしていることがわかりました。

具体的には主に3パターン。

| パターン | 何をしているか |
|---|---|
| カメラ視点の取得 | プレイヤーカメラの位置・向きを使ってヒットスキャンや飛び道具の起点を決める |
| ロック対象の解決 | カーソルが当たっているアクターをロックターゲットに使う |
| ロール方向の入力 | 直前のスティック入力からロール方向を読み取る |

これらは**プレイヤー操作前提の設計**なので、AIコントローラーから同じアビリティを起動しても正しく動きません。今のままだとAI用にアビリティクラスを丸ごと複製する必要が出てきます。それは避けたい。

![プレイヤーコントローラー依存の3パターン](/marcode_aulinux/assets/img/posts/2026-02-15/02-coupling-patterns.svg)
_アビリティからプレイヤーコントローラーへの直接依存3パターン（プレースホルダー）_

## 方針：プレイヤー依存部分を「差し替えられる口」に変える

コードを書き換えるのではなく、**プレイヤー固有の処理を仮想関数やユーティリティ関数に置き換えて、AI側とプレイヤー側で実装を差し替えられるようにする**方針です。

### カメラ視点の取得（最もシンプルな修正）

ヒットスキャン・飛び道具・テレポートなど複数のアビリティで、プレイヤーカメラの位置を起点として使っています。

これを「プレイヤーならカメラ位置、そうでなければキャラクターの目の位置」を返す共通ヘルパーに一本化します。AIはカメラを持たないので自動的に目の位置にフォールバックしますが、AIは常にターゲット方向を向いているので実用上は問題ありません。

![視点取得ヘルパーの分岐ロジック](/marcode_aulinux/assets/img/posts/2026-02-15/03-viewpoint-helper.svg)
_プレイヤーはカメラ位置、AIはキャラクターの目の位置にフォールバックし、同一の出力を返す（プレースホルダー）_

### ロック対象の解決（仮想フック化）

`GA_LockAim`はカーソルのハイライト対象をロックターゲットにしています。この処理を仮想関数に切り出し、**デフォルトは`nullptr`を返す**ようにします。

- **プレイヤー用子クラス** → カーソルのアクターを返すように上書き
- **AI側** → `nullptr`のままなので、AIがすでに持っている戦闘ターゲット情報にフォールバック

AIは別のシステムで戦闘ターゲットを常に管理しているので、追加実装なしで正しいターゲットが使われます。

### ロール方向の入力（同じく仮想フック化）

`GA_FocusRoll`はスティック入力の向きを読んでロール方向を決めています。これも仮想関数に置き換え、**デフォルトは「明示的な方向なし」を返す**ようにします。

- **プレイヤー用子クラス** → コントローラーから入力方向を取得
- **AI側** → フォールバックとしてキャラクターの移動ベクトルを使用（AIが移動を制御しているので自然に正しい方向になる）

## 結果：クラスの複製なしで共用できる

3つの修正はすべて**追加的な変更**です。既存のプレイヤー側の動作には一切触れず、AI用に新しいアビリティクラスを作ることもしません。

```
プレイヤー  →  BP_GA_FocusRoll_Player  (プレイヤー向け子クラス)
                      ↑ 継承
AI          →  GA_FocusRoll            (ベースクラスをそのまま使用)
```

コンボ技については、AI側は1アクティベーションごとに1スイングを行う方式でとりあえず十分です。コンボの連鎖が必要になってきたタイミングで対応します。

![共有クラス構造の結果](/marcode_aulinux/assets/img/posts/2026-02-15/04-shared-class-result.svg)
_ベースクラスをAIが直接使い、プレイヤー用子クラスがフックを上書きする構成（プレースホルダー）_

## 実装順序

一気にやらず段階的に進めます。

1. **カメラ視点ヘルパーの追加** → ヒットスキャン・飛び道具・テレポート系を一括で修正（プレイヤーへの影響ゼロ）
2. **仮想フックの追加** → ロール方向・ロックターゲットをプレイヤー子クラスに分離
3. **AIタスクの整備** → 各アビリティを起動するAI側の設定をエディタ上で組む

コンパニオンが主人公と同じ技を使えるようになると、ゲームプレイの幅が一気に広がります。土台を丁寧に作っておくことで、後からキャラクターを増やす際のコストも下げられます。
